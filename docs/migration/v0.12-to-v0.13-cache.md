# Migration Guide: Cache Versioning (v0.12.x to v0.13.x)

This guide explains the cache versioning changes introduced in v0.13.0 and provides migration strategies for existing installations.

## What Changed

### Before (v0.12.x and earlier)
- Cache entries stored only content hash and analysis results
- No way to detect if cached analysis was outdated due to code changes
- Same cache entry used indefinitely as long as file content unchanged

### After (v0.13.0)
- Cache entries now include version fields:
  - `schema_version` - Tracks CachedAnalysis structure changes
  - `metadata_version` - Tracks metadata extraction logic changes
  - `semantic_version` - Tracks semantic analysis logic changes
- Automatic staleness detection when application version changes
- New CLI commands for cache management

## Impact on Existing Installations

When upgrading to v0.13.0:

1. **Existing cache entries are preserved** - Your cache directory is not modified during upgrade
2. **Legacy entries marked as v0.0.0** - Entries without version fields are identified as "legacy"
3. **Legacy entries work but may be stale** - They can still be used but may have outdated analysis

## Migration Strategies

### Strategy 1: Do Nothing (Recommended for Most Users)

**Best for:** Users who want minimal disruption and are okay with gradual cache refresh

```bash
# Check your current cache status
agentic-memorizer cache status

# Continue using the daemon normally
agentic-memorizer daemon start
```

**What happens:**
- Legacy cache entries continue working immediately
- Files are re-analyzed when their content changes
- Cache gradually refreshes over time as you modify files
- No immediate API cost or rebuild time

**Pros:**
- Zero downtime
- No additional API calls
- Gradual transition

**Cons:**
- Inconsistent analysis quality between old and new entries
- Legacy entries may lack newer analysis features

### Strategy 2: Clear Stale Entries Only

**Best for:** Users who want consistent analysis without full rebuild

```bash
# Check what will be cleared
agentic-memorizer cache status

# Clear only legacy/stale entries
agentic-memorizer cache clear --stale

# Rebuild index to re-analyze affected files
agentic-memorizer daemon rebuild
```

**What happens:**
- Only legacy (v0.0.0) entries are removed
- Current version entries are preserved
- Affected files will be re-analyzed during rebuild
- Files with current cache entries skip re-analysis

**Pros:**
- Targeted cleanup
- Preserves recent work
- Moderate API cost

**Cons:**
- Requires daemon rebuild
- API calls for all previously cached files

### Strategy 3: Full Cache Rebuild

**Best for:** Users who want completely fresh analysis with latest logic

```bash
# Clear entire cache and rebuild
agentic-memorizer cache clear --all
agentic-memorizer daemon rebuild

# Or use the combined flag
agentic-memorizer daemon rebuild --clear-old-cache
```

**What happens:**
- All cached entries are removed
- Every file will be re-analyzed from scratch
- All entries will have current version numbers

**Pros:**
- Cleanest state
- Consistent analysis across all files
- Takes advantage of all latest improvements

**Cons:**
- Highest API cost
- Longest rebuild time
- May hit rate limits on large repositories

## New CLI Commands

### cache status

Shows cache statistics including version distribution:

```bash
$ agentic-memorizer cache status

Cache Status
============

Location:       /Users/you/.agentic-memorizer/.cache
Current Version: v1.1.1

Statistics
----------
Total Entries:  1234
Total Size:     45.6 MB
Legacy Entries: 890

Version Distribution
--------------------
  v0.0.0: 890 (legacy)
  v1.1.1: 344 (current)
```

### cache clear

Clears cache entries with two modes:

```bash
# Clear only legacy/stale entries
agentic-memorizer cache clear --stale

# Clear all entries
agentic-memorizer cache clear --all
```

### daemon rebuild --clear-old-cache

Combines stale cache clearing with index rebuild:

```bash
# Clear stale entries before rebuilding
agentic-memorizer daemon rebuild --clear-old-cache
```

## Understanding Version Numbers

Cache versions follow the format `vX.Y.Z`:

| Component | Meaning | When It Changes |
|-----------|---------|-----------------|
| X (Schema) | CachedAnalysis structure | Field additions/removals to cache structure |
| Y (Metadata) | Metadata extraction | Changes to file metadata extraction |
| Z (Semantic) | Semantic analysis | Changes to AI prompts or analysis logic |

**Staleness Rules:**
- Schema mismatch (X differs) = Always stale
- Metadata behind (Y lower) = Stale, needs newer metadata
- Semantic behind (Z lower) = Stale, needs newer analysis
- Future versions = Not stale (forward compatible)

## Troubleshooting

### "Many legacy entries" warning

If `cache status` shows many legacy entries:

```bash
# Option 1: Clear only legacy entries
agentic-memorizer cache clear --stale

# Option 2: Live with gradual refresh (no action needed)
```

### High API usage after upgrade

If you notice higher API usage:

1. Check cache hit rate in daemon logs
2. Consider clearing old versions if cache is mostly legacy
3. Rate limits are enforced (default 20 calls/min)

### Cache directory location

The cache is stored at the path configured in `analysis.cache_dir`:

```bash
# Check your config
agentic-memorizer config validate

# Default location
~/.agentic-memorizer/.cache/summaries/
```
